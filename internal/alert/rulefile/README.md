# 告警规则与规则组说明

## 当前策略：只记录分组，不写文件、不挂载

规则与规则组**仅在本系统内按分组记录**（数据库），用于：

- 从 Prometheus 同步来的规则按规则组展示、管理；
- 在本系统创建/编辑的规则归属到某个规则组；
- 告警规则列表、筛选、分组展示。

**默认不写本地规则文件、不依赖 Prometheus 挂载**。多实例部署（多台 EC2）时无需共享存储或挂载，规则与分组数据在 DB 中一致即可。

- **规则目录**：未设置环境变量 `ALERT_RULE_DIR` 时，不写入任何规则文件，不调用 Prometheus `/-/reload`。
- 若你仍需要“写本地文件 + Prometheus 读该目录”的旧方式，可设置 `ALERT_RULE_DIR`（例如 `/etc/prometheus/rules`），并自行保证 Prometheus 能读到该目录（单机或共享存储）。此方式为可选，多实例时需自行处理共享与一致性。

## 规则组（分组）含义

- **规则组**：在 DB 中表示一组告警规则的逻辑分组（名称、描述、关联数据源等）。当前策略为仅记录、不挂载，故**规则组不再展示或编辑「规则文件路径」（File）**；若日后启用写文件（设置 `ALERT_RULE_DIR`），后端会按需生成或使用规则组对应文件路径。
- **规则**：每条告警规则归属一个规则组（`group_id`），便于在 UI 中按组管理、筛选。
- 从 Prometheus 同步时，会按 Prometheus 返回的 group 名称创建或匹配本系统的规则组，并把规则挂到对应组下。

### 为什么规则组必须属于某个数据源？

规则组在设计上**必须归属一个数据源**（`source_id`），不能做成“全局规则组”再选数据源，原因：

- 每个数据源对应一个 Prometheus（或 VictoriaMetrics）实例，规则和规则组都是**按实例维度**存在的；同步时是从该实例拉取 group 列表与规则，写入时也是按实例/数据源写对应规则文件。
- 若规则组与数据源解耦，则“某条规则属于哪个实例”会变得模糊，同步、写入、归属关系都难以维护。
- 因此：**先选数据源，再选该数据源下的规则组**（二级联动）是合理且必要的。新建告警规则时需先选数据源，规则组下拉框只展示该数据源下的规则组；告警规则列表也会展示每条规则所属的规则组。

## 可选：写文件 + Prometheus 挂载（仅当设置 ALERT_RULE_DIR 时）

仅当部署时设置了 `ALERT_RULE_DIR` 时，本系统才会：

1. 在该目录下按规则组的 `File` 写入 YAML 规则文件；
2. 在用户点击「重新加载」时向数据源发送 `POST /-/reload`。

此时需自行保证 Prometheus 的 `rule_files` 包含该目录，且 Prometheus 能读到（同机或共享卷）。多实例时需共享该目录或只由单实例写入，详见历史提交或注释。

**结论**：日常使用建议不设置 `ALERT_RULE_DIR`，只做记录分组即可；挂载方案按需可选。
